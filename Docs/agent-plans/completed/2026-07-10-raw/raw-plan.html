Overview

The goal is to build SwiftRAW, a Swift library that reads camera RAW files by wrapping LibRAW. The wrapper exposes RAW files through a clean, Swift-native, read-only API — structured metadata (camera / lens identification, exposure, image geometry, color data, GPS / timestamp) plus access to the unpacked raw sensor (Bayer) buffer. It intentionally does not demosaic to RGB or extract thumbnails, mirroring the read-only nature of the author's SwiftFITS and SwiftXISF libraries.

LibRAW is a C++ library. Per the source document, Swift will talk to it directly through C++ interop: a Clang module exposes the C++ LibRaw class, and the Swift layer drives its lifecycle (open_file / unpack / recycle) and reads structured data off its imgdata member. The plan follows the reference projects' architecture and conventions as closely as applicable: a core RAWFile reference type, a RAWError enum, one type per file organized into subfolders, exhaustive SwiftDoc comments, and a broad Swift Testing suite that discovers fixture files at the repository root. Both the Xcode project and the SwiftPM manifest must build and test green, matching the existing CI matrix.

Questions & answers

Q1 — How should the Swift layer bind to LibRAW under the hood?

The C++ LibRaw class via Swift/C++ interop. Drive lifecycle through the C++ class and read metadata off its imgdata member, matching the stated goal of "using C++ directly from Swift." The plain-C libraw_* functions remain available as a fallback for spots where interop is awkward, but the C++ class is the primary binding surface.

Q2 — What should the wrapper's public API expose?

Metadata + raw sensor data only. Open / unpack a RAW file and expose structured metadata (camera & lens, exposure, sizes, CFA pattern, color data, GPS / timestamp) plus access to the raw Bayer / sensor buffer. No demosaicing to RGB, no thumbnail extraction — the closest analog to the read-only reference parsers.

Q3 — How should the test suite obtain RAW fixture files?

Real RAW samples, provided on demand. Mirror SwiftXISF: a Test Files/ directory at the repository root, discovered via #filePath under SwiftPM and via the test bundle under Xcode. The user will supply real samples (e.g. .CR2 / .NEF / .ARW / .DNG); the suite is structured to test broadly against them.

Q4 — How should the Swift package link the vendored LibRAW static library?

Recommended: system-library Clang module + linker unsafeFlags (see Notes → SPM linkage decision). This reuses the universal .a already committed to the repo, matches how the Xcode target already links, and its only real caveat — consumers must depend via a branch / revision rather than a versioned tag — is already how SwiftFITS and SwiftXISF are consumed. An xcframework binary target is recorded as an alternative if versioned consumption later becomes a requirement.

Notes

Current state of the repository

  • The Xcode target is already wired for LibRAW: HEADER_SEARCH_PATHS = LibRAW/include, LIBRARY_SEARCH_PATHS = LibRAW/lib, OTHER_LDFLAGS = -lraw_r, and SWIFT_OBJC_INTEROP_MODE = objcxx (C++ interop enabled). What is missing is a Clang module map so Swift can import LibRAW.
  • The SwiftPM manifest has no LibRAW wiring at all — no module, no C++ interop mode, no linkage. This is the largest build-system gap and is addressed first (Milestone 1).
  • Source scaffolding is a single empty SwiftRAW/SwiftRAW.swift and a placeholder test. Both will be replaced by the subfolder-organized structure below.
  • Both .a variants are universal (x86_64 + arm64). Use the thread-safe libraw_r variant (already referenced by OTHER_LDFLAGS).
  • CI (.github/workflows/ci-mac.yaml) already builds both the Xcode project (Debug test + Release build) and SwiftPM (swift build / swift test, debug + release). Every milestone must keep both paths green.

Conventions to follow (from SwiftFITS / SwiftXISF and the global working agreement)

  • MIT license header block at the top of every source file (2026 copyright), as in the existing files.
  • One public type per file; organize into subfolders (proposed: Core/, Image/, Metadata/, Color/, Lens/, Extensions/).
  • A RAWError enum conforming to LocalizedError, CustomStringConvertible, Sendable, with a "RAW Error: …"-style description — mirroring XISFError.
  • Explicit public; let over var; no force-unwrap; early returns; prefer map/filter/forEach over raw loops; aligned =; blank line between calls and assignments.
  • Full SwiftDoc comments on all members, including private ones.
  • Tests use Swift Testing (import Testing, @Test, #expect, #require), with a test-only TestError and a TestUtilities fixture discoverer that mirrors SwiftXISF's #filePath / bundle split.
  • The Xcode project and Package.swift are maintained together and kept in sync at every step; prefer xcodebuild for building/testing, but keep swift build/swift test working too.
  • Do not reference absolute paths to SwiftFITS / SwiftXISF anywhere in the produced code or docs.

C++ interop specifics

  • libraw.h is C++ and pulls in <limits.h>, <memory.h>, etc., then declares both the extern "C" API and the class LibRaw. The module map must mark the module requires cplusplus.
  • Xcode uses SWIFT_OBJC_INTEROP_MODE = objcxx; SwiftPM needs the equivalent .interoperabilityMode(.Cxx) Swift setting on the SwiftRAW target. Both must agree.
  • Interop risk areas to validate early: importing a class with virtual methods and non-trivial construction, reading nested C structs off imgdata, and mapping char[] fields / C strings to Swift String. Where the C++ class is awkward, the flat libraw_* C functions are an acceptable fallback.
  • The LibRaw instance owns heap buffers; the Swift wrapper must keep it alive for as long as any exposed raw-buffer view is in use, and call recycle() on deinit. The wrapper is a reference type and is not Sendable (same stance as FITSFile / XISFFile).

SPM linkage decision (recommendation for Q4)

Use a system-library Clang module (a shared module.modulemap that both Xcode and SwiftPM consume) plus linker unsafeFlags (-L LibRAW/lib -lraw_r) and header include flags on the SwiftRAW target. Rationale: it reuses the committed universal .a, matches the Xcode target's existing linkage, and requires no repackaging. The one caveat — unsafeFlags prevents consumption via a versioned tag, so dependents must use branch:/revision: — is already the distribution model for SwiftFITS/SwiftXISF. If clean versioned consumption is needed later, repackage LibRAW as an xcframework binary target; that alternative is deferred, not chosen. The exact module-map placement and search-path wiring is finalized and verified in Milestone 1.

Dependencies & risks

  • Test fixtures are user-supplied. Milestones that assert on real metadata / buffers cannot be fully validated until sample RAW files are provided. The fixture-discovery mechanism and error-path tests are built regardless so the suite is runnable from the start.
  • Maker-note breadth. LibRAW exposes very large, camera-specific maker-note structs. Scope keeps a curated, cross-vendor subset (identification, exposure, lens, color, GPS); exhaustive per-vendor maker notes are out of scope unless requested. Flagged in Milestone 5.
  • C++ interop maturity. If a specific construct fails to import, fall back to the C API for that call. Surface any blocker early rather than guessing.

The smoke test therefore instantiates via the heap-based C API (libraw_init / libraw_close) — the Q1 fallback for awkward interop. This shapes M2: RAWFile cannot hold LibRaw as a Swift value member; it must keep the instance on the heap (C API context, or a C++ shim).

Critical bug found & fixed during review: LibRaw::open_buffer wraps the caller's pointer in a datastream that stores only a bare unsigned char *buf (verified in libraw_datastream.h) and reads it lazily during unpack — it does not copy. Fixed by copying the bytes into a stable UnsafeMutableRawBufferPointer the file owns for its lifetime, freed in deinit (after recycle()) and on every throwing path. Full runtime validation of the read path awaits real fixtures (the openFixtureFromData test covers it); the fix is correct by construction and confirmed by re-review.

Implementation notes

The buffer-access API design (breadth and lifetime model) was proposed and confirmed by the owner before implementing: a focused scope (16-bit Bayer only, with a descriptor reporting the other layouts) offering both a copy and a borrowed view.

  • RAWSensorData (Image/): a value-type descriptor of the unpacked buffer — width/height/pitch from libraw_image_sizes_t, plus a Layout enum (none/bayer/color3/color4 and the three float variants) derived from whichever libraw_rawdata_t pointer LibRAW populated, with derived componentCount, isFloatingPoint, and hasBuffer. So a consumer always knows the layout, even for the uncommon multi-channel/float variants that have no dedicated data accessor.
  • RAWFile.rawImage: [UInt16]?: a Swift-owned copy of the 16-bit single-channel Bayer buffer, nil when unavailable (not unpacked, or a non-Bayer layout).
  • RAWFile.withRawImage(_:): a zero-copy, closure-scoped borrowed view (UnsafeBufferPointer<UInt16>) into LibRAW-owned memory, valid only for the call and tied to the file's lifetime; documented not to escape. rawImage is implemented on top of it (withRawImage { Array( $0 ) }).
  • RAWFile.sensorData: built via withUnsafePointer( to: &…imgdata.rawdata ), reading only the small buffer-pointer fields — it never copies the ~130 KB rawdata struct (which embeds a full color-data with the 128 KB tone curve).

Buffer sizing: the 16-bit Bayer element count is raw_height × (raw_pitch / 2) (raw_pitch being the byte pitch of the ushort raw_image), guarded on height > 0 and pitch > 0. Not-unpacked and no-buffer cases return nil/.none.
